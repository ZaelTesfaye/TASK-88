# Mock and Stub Disclosure

This document discloses how mock and stub components are used in the Multi-Org Data & Media Operations Hub.

---

## Principle: No External Service Dependencies

This application is designed for **LAN-only operation**. It does not call any third-party cloud APIs, SaaS endpoints, or external services in production. All integrations target LAN-accessible resources (local folders, network shares, local databases).

---

## Third-Party Service Status

| Service category       | Real integration | Mock/stub present | Notes                          |
|------------------------|:----------------:|:-----------------:|--------------------------------|
| Cloud storage          | No               | No                | All file I/O is local/LAN      |
| External APIs          | No               | No                | No outbound API calls           |
| Email/notification     | No               | No                | Report delivery is file-based   |
| Biometric auth         | No               | Stub only         | Gated by `ENABLE_BIOMETRIC`     |
| External databases     | Yes              | No                | `DatabaseConnector` opens real `database/sql` connections for pull     |

---

## Biometric Authentication Stub

The application includes a biometric authentication code path that is **disabled by default**.

| Environment variable | Default | Effect when `true`               |
|----------------------|---------|----------------------------------|
| `ENABLE_BIOMETRIC`   | `false` | Enables biometric auth endpoints |

When `ENABLE_BIOMETRIC=false` (the default in all environments including production), biometric endpoints are not registered and cannot be reached. This is a placeholder for future hardware integration, not an active mock.

---

## Database Connector

The `DatabaseConnector` (type `database`) in `backend/internal/ingestion/connector.go` implements the full `Connector` interface with real `database/sql` connectivity:

- `ValidateConfig` -- validates required config keys and opens a test connection to verify reachability
- `HealthCheck` -- opens a real connection and pings the database
- `Pull` -- executes cursor-based incremental queries against the configured table or query; supports backfill (empty cursor = full scan) and incremental pull (cursor = last-seen value)
- `AcknowledgeCheckpoint` -- no-op (checkpoint persistence is managed by the ingestion engine)

## Production-Safe Defaults

All feature flags and configuration defaults are set for safe production operation:

| Variable            | Default value   | Rationale                                       |
|---------------------|-----------------|--------------------------------------------------|
| `APP_ENV`           | `production`    | Strictest behavior by default                    |
| `ENABLE_BIOMETRIC`  | `false`         | Stub not exposed                                 |
| `ENABLE_TLS`        | `false`         | TLS opt-in for development convenience           |
| `ALLOWED_HOSTS`     | LAN CIDRs only  | Egress guard blocks non-LAN IPs                  |
| `LOG_LEVEL`         | `info`          | Production-appropriate logging                   |

---

## Test Mocks

In the test suites (`backend/tests/` and `frontend/src/tests/`), standard test doubles are used:

- **Backend**: Go test helpers create in-memory or test-scoped database instances. No external mocking frameworks are used. JWT tokens for test users are generated by `helpers_test.go`.
- **Frontend**: Vitest provides built-in mocking for API calls. Store tests use mocked HTTP responses.

These test mocks are confined to test files and are never compiled into or reachable from the production binary.
